The Making of Polymatic — Architecture Diagrams

PART 2 · FASTAPI BACKEND · SIX DIAGRAMS FOR A LAY READER

01
The Big Picture
Where does data come from, where does it go, and what sits in the middle?
Summary: Data enters from five external sources on the left, gets processed inside one Python server process in the middle, and arrives at the analyst's browser on the right. The dashed box is the entire backend — one process, one machine. Purple lines are live WebSocket pushes; grey dashed lines are regular HTTP requests.
02
One Process, Three Workloads
Everything runs inside a single uvicorn server — no separate worker machines, no message broker.
Summary: Most backend systems use separate machines for web serving, background jobs, and real-time messaging. Polymatic runs all three inside one Python process on one server. This is simpler to deploy and operate — but it requires every component to be written in a way that doesn't block the others. That's what "async-native" means in practice.
03
The Request Lifecycle
What actually happens between "the browser asks for feed items" and "the browser receives them."
Plain English: The browser sends a question. The router figures out what's being asked. The service builds the database query. PostgreSQL looks up the rows. Pydantic shapes the data into clean JSON (hiding internal fields like raw_payload). The JSON goes back to the browser. Five steps, typically under 20 milliseconds.
04
The WebSocket Flow
How a ship going dark in the Persian Gulf reaches an analyst's screen in under a second.
Two paths to the analyst's screen: The top row shows what happens every 60 seconds when the scheduler wakes up — fetch, deduplicate, write to database, then push to every connected analyst simultaneously. The bottom row shows what happens the moment a new analyst opens the dashboard — they immediately receive the last 50 items as a snapshot, then switch into live-push mode for all subsequent updates.
05
The Database Schema
Six tables, what each one stores, and how they relate — without the SQL.
Reading this diagram: Each coloured box is a database table. Lines between boxes mean "this table references that one." feed_sources → feed_items means one source (e.g. Bellingcat) can have many feed items. sentiment_questions → sentiment_aggregates has exactly 3 rows per question — one for each time window (live / trend / baseline). The raw_payload column in feed_items (shown in orange) stores the original API response but is never sent to the browser.
06
The Async Event Loop
How one Python process handles hundreds of simultaneous requests without blocking — the core idea behind "async-native."
The core idea in one sentence: Instead of one request monopolising the server until it finishes, every operation that involves waiting (database query, external API call, file write) uses await — which means "pause me here and go do something else until the result arrives." The event loop keeps cycling, picking up whichever task has a result ready. One Python thread ends up handling hundreds of simultaneous requests, ingestion jobs, and WebSocket connections without any of them blocking the others.
