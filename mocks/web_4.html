THE PROBLEM
THREE WORKLOADS — ALL ASYNC — ALL NEED TO COMPOSE IN ONE PROCESS
REST API
Serves the frontend's DataProvider interface — feed items, events, sentiments, markets
WebSocket Layer
Pushes real-time feed items and sentiment updates to every connected analyst browser
Ingestion Pipeline
Polls LiveUAMap, Bellingcat, Polymarket, Kalshi on per-source schedules — no idle waiting
All three are async by nature. All three need to compose cleanly in a single process — no message broker overhead, no separate worker fleet to operate.

THE TRADEOFF I'M ACCEPTING
⚠
TRADEOFF
NO BUILT-IN ADMIN PANEL
FastAPI ships with no Django-style admin. Every data management operation — configuring feed sources, editing credibility scores, auditing classification results — needs either a separate admin UI or direct database access during the early phases.

For now, psql and a seeded database gets us to launch. I'll address this in Part 4 with a minimal internal tooling layer. It's a real cost — and it's the right tradeoff for a team of one shipping a real-time intelligence platform that needs to stay operationally simple.
